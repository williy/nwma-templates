# NWMA Templates: IT資産価値管理方式 (Net Worth Management Architecture)

> **「SLAの自動執行までを設計のゴールとする」**

NWMAは、属人化した設計判断を構造化し、ITシステムを「管理負債」ではなく、継続的に価値を生む「資産」として維持するための設計フレームワークです。

## 2. アーキテクチャの階層定義

NWMAは、以下の 3 つのレイヤーによってシステムの整合性を自律的に維持します。

* **NWMA (Standard / 設計基準)**
全体を貫く共通規格。要件（A）・構造（B）・試験（C）をセットで管理し、長期間にわたって「設計意図」を継承できる資産価値を確立します。
* **IWMA (Foundation / 基盤層)**
インフラ基盤。Terraform 等を用い、クラウドベンダーの仕様変更や製品寿命に左右されない、柔軟で堅牢な境界定義を行います。
* **PWMA (Platform / 実行層)**
プラットフォーム。Ansible 等を用い、設計通りの状態を維持します。構成ドリフトを抑止し、常に「あるべき状態」への復元を可能にします。

---

## 3. 永続化の原則：環境とデータの分離

NWMAは「実行環境（OS/MW）」と「保持すべき資産（データ）」を論理的に分離する、クリーンなアーキテクチャを推進します。

* **環境の再生成:** 異常検知時には自動的に再デプロイを行い、常に正常な初期状態へ復帰させます。
* **データの保護:** データベースやストレージは、再生成の対象外として厳密に保護します。環境を何度作り直しても、蓄積されたデータ資産は影響を受けません。

これにより、バックアップ対象を「純粋なデータ層」に限定し、RTO（目標復旧時間）の確実な保証を実現します。

---

## 4. 管理シート：A/B/C Sheets

設計情報を 3 つの役割に分類し、完全なトレーサビリティを確保します。

| シート名 | 役割 | 出力 / 対象 |
| --- | --- | --- |
| **A-Sheet (Requirement)** | 設計判断の根拠 | 投資対効果、SLA/SLOの定義 |
| **B-Sheet (Structure)** | 構成の正本 | IaC (Terraform / Ansible), 責任境界 |
| **C-Sheet (Verification)** | 設計の妥当性証明 | 自動検証スクリプト, 構成逸脱率の計測 |

---

## 5. ディレクトリ構造

管理対象ごとにディレクトリを分離し、運用ワークフローを明確化します。

```text
nwma-templates/
├── manifesto/          # NWMAの基本理念・定義（core_philosophy.yaml）
├── A-Sheets/           # 要件定義・設計判断理由の記録
├── B-Sheets/           # 構成定義と自動執行スクリプト
│   ├── IWMA/           # インフラ基盤定義（Terraform等）
│   └── PWMA/           # プラットフォーム構成（Ansible等）
├── C-Sheets/           # 自動検証・テストスクリプト
├── examples/           # 構成例（K3s on Ubuntu等）
└── templates/          # 各種ドキュメント・設定の雛形

```

---

## 6. NWMAを採用するメリット

* **責任境界の明確化:** B-Sheetによって担当領域を可視化し、運用の不透明さを解消します。
* **SLAの自動維持:** 監視と連動し、異常発生時に「環境の再デプロイ」を自動で行うことで、可用性を高めます。
* **データ構造化:** 設計判断がYAML形式等で構造化されているため、将来的なAIによる解析や自動保守にもスムーズに対応可能です。
